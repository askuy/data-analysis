# CTO 产研数据分析能力 (Skill)

## 角色定义

作为公司CTO，需要具备从产研周报、项目文档、故障报告等多维度数据中快速提取关键信息，做出技术决策和资源调配的能力。

---

## 一、核心分析维度

### 1.1 私有化交付管理

**关注重点：**
- POC项目进展和转化率
- 交付部署的时间线和阻塞点
- 版本升级的风险和客户反馈
- 定制化需求的成本评估

**关键问题：**
- 哪些POC项目已超期？原因是什么？
- 当前有多少客户在等待升级？
- 定制化需求占比多少？是否影响标准产品迭代？
- 交付团队的负载情况如何？

**决策支持：**
- 优先级排序：根据客户价值和紧急程度调整交付顺序
- 资源调配：识别瓶颈并分配人力
- 风险预警：提前识别可能延期的项目

### 1.2 产品线健康度

**关注产品：**
| 产品线 | 战略重要性 | 关注指标 |
|--------|-----------|----------|
| Drive | 高 | 功能完成度、客户采纳率 |
| 应用表格 | 高 | 性能优化、重构进度 |
| 轻文档 | 中 | 稳定性、重构进度 |
| 专业文档 | 中 | 兼容性、渲染能力 |
| 表格计算 | 中 | 灰度进度、性能提升 |
| 极速SDK | 中 | 与标准SDK集成进度 |
| 协同编辑 | 高 | 稳定性、并发能力 |
| AI能力 | 高 | 知识库、ChatOffice进展 |

**关键问题：**
- 各产品线的开发进度是否符合预期？
- 技术债务是否在持续增加？
- 哪些产品线需要增加投入？
- 是否有产品线应该暂停或下线？

### 1.3 系统稳定性与故障管理

**故障分级：**
| 级别 | 定义 | 响应要求 |
|------|------|----------|
| P0 | 核心功能不可用，影响大量用户 | 立即响应，全员参与 |
| P1 | 重要功能异常，部分用户受影响 | 1小时内响应 |
| P2 | 次要功能问题，少量用户受影响 | 当天处理 |

**关注指标：**
- 月度故障数量和趋势
- 故障恢复时间(MTTR)
- 故障按客户分布
- 重复故障率
- 根因分析完成率

**关键问题：**
- 故障数量是上升还是下降？
- 哪些客户故障频发？根因是什么？
- 是否有系统性问题导致的故障模式？
- 预防措施是否有效？

### 1.4 客户健康度评估

**评估维度：**
```
客户健康度 = f(活跃度, 故障率, 满意度, 续费风险)
```

**健康状态分类：**
| 状态 | 特征 | 行动 |
|------|------|------|
| 健康 (✅) | 故障率<20%，正常使用 | 保持现状 |
| 关注 (⚠️) | 故障率20-50%，有抱怨 | 主动跟进 |
| 危险 (🔴) | 故障率>50%，可能流失 | 优先处理 |

**关键问题：**
- 哪些客户需要重点关注？
- 高价值客户的满意度如何？
- 是否有客户流失风险？
- 客户的定制需求是否可持续？

### 1.5 团队与资源管理

**关注领域：**
- 人员变动：离职、入职、调岗
- 团队负载：是否有瓶颈
- 技能缺口：哪些能力需要补充
- 组织架构：是否需要调整

**关键问题：**
- 关键岗位是否有备份？
- 哪些团队压力过大？
- 外包资源使用是否合理？
- 是否需要招聘或培养特定技能？

### 1.6 成本与效率

**关注指标：**
- 服务器成本月度趋势
- 人力成本分配
- 私有化项目利润率
- 技术债务偿还投入

**关键问题：**
- 服务器成本是否在可控范围？
- 哪些优化措施已产生效果？
- 私有化项目是否盈利？
- 资源投入产出比是否合理？

## 二、分析框架

### 2.1 周度分析流程

```
1. 快速扫描
   ├── 本周重大故障
   ├── 关键客户状态
   └── 阻塞性问题

2. 深度分析
   ├── 产品线进展 vs 计划
   ├── 资源使用情况
   └── 风险识别

3. 决策输出
   ├── 需要介入的事项
   ├── 资源调配建议
   └── 下周关注重点
```

### 2.2 月度分析框架

```
┌─────────────────────────────────────────────────────────┐
│                    月度健康报告                          │
├─────────────────────────────────────────────────────────┤
│ 1. 业务指标                                              │
│    - 私有化交付完成数                                    │
│    - 新签客户数                                          │
│    - 客户满意度                                          │
├─────────────────────────────────────────────────────────┤
│ 2. 技术指标                                              │
│    - 故障数量和MTTR                                      │
│    - 产品发布数                                          │
│    - 技术债务偿还进度                                    │
├─────────────────────────────────────────────────────────┤
│ 3. 团队指标                                              │
│    - 人员稳定性                                          │
│    - 关键项目进度                                        │
│    - 团队满意度                                          │
├─────────────────────────────────────────────────────────┤
│ 4. 成本指标                                              │
│    - 服务器成本                                          │
│    - 人力成本                                            │
│    - 项目利润                                            │
└─────────────────────────────────────────────────────────┘
```

### 2.3 风险识别矩阵

| 风险类型 | 识别信号 | 应对策略 |
|----------|----------|----------|
| 客户流失 | 故障频发、抱怨增加 | 主动沟通、优先修复 |
| 项目延期 | 连续周报提及阻塞 | 资源倾斜、调整计划 |
| 人员流失 | 关键人员变动信号 | 提前沟通、知识转移 |
| 技术风险 | 新技术引入问题 | 灰度控制、回滚预案 |
| 成本超支 | 资源使用超预期 | 优化方案、预算调整 |

## 三、关键指标(KPI)体系

### 3.1 交付效率指标

| 指标 | 计算方式 | 目标值 |
|------|----------|--------|
| POC转化率 | 签约数/POC数 | >60% |
| 交付周期 | 从签约到上线天数 | <30天 |
| 升级成功率 | 一次成功升级/总升级 | >90% |
| 客户满意度 | 交付评分平均值 | >4.5/5 |

### 3.2 稳定性指标

| 指标 | 计算方式 | 目标值 |
|------|----------|--------|
| 月度P0故障 | P0故障数量 | <2 |
| MTTR | 故障恢复平均时间 | <30min |
| 故障重现率 | 重复故障/总故障 | <10% |
| SLA达成率 | 达标天数/总天数 | >99.9% |

### 3.3 产品进度指标

| 指标 | 计算方式 | 目标值 |
|------|----------|--------|
| 需求完成率 | 完成需求/计划需求 | >80% |
| 灰度成功率 | 成功灰度/总灰度 | >95% |
| Bug修复周期 | 从发现到修复天数 | <3天 |
| 技术债务比 | 债务工作量/总工作量 | <20% |

### 3.4 成本效率指标

| 指标 | 计算方式 | 目标值 |
|------|----------|--------|
| 服务器成本 | 月度云资源费用 | <50万 |
| 人效比 | 收入/人力成本 | >3 |
| 私有化毛利 | (收入-成本)/收入 | >40% |

## 四、数据来源

### 4.1 主要数据源

| 数据源 | 类型 | 更新频率 | 用途 |
|--------|------|----------|------|
| 产研周报 | Markdown | 每周 | 项目进展、故障、人员 |
| 故障报告 | 文档链接 | 实时 | 故障分析、根因定位 |
| Jira/项目管理 | 系统数据 | 实时 | 任务进度、工时 |
| 财务数据 | 报表 | 每月 | 成本、收入分析 |
| 客户反馈 | 记录 | 实时 | 满意度、需求 |

### 4.2 数据质量要求

- **完整性**: 周报需覆盖所有产品线和重点客户
- **及时性**: 数据需在事件发生后24小时内记录
- **准确性**: 关键数据需可追溯验证
- **一致性**: 使用统一的术语和分类标准

## 六、使用指南

### 6.1 快速分析命令

```bash
# 生成执行摘要
python analyze.py summary

# 搜索特定客户的所有记录
python analyze.py client [客户名]

# 搜索关键词
python analyze.py search [关键词]

# 查看故障分析
python analyze.py incidents

# 查看产品线进度
python analyze.py products

# 导出完整分析
python analyze.py export
```

### 6.2 常用分析场景

| 场景 | 操作 | 目的 |
|------|------|------|
| 周会准备 | `summary` | 快速了解整体状况 |
| 客户问题 | `client X` | 查看特定客户历史 |
| 故障复盘 | `search 故障` | 找出故障模式 |
| 产品规划 | `products` | 了解各产品线状态 |
| 趋势分析 | 查看JSON导出 | 深度数据分析 |
