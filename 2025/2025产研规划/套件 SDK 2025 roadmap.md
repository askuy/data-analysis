# 背景

目前石墨套件代码中包含了大量非套件核心逻辑，并且存在许多硬编码的功能配置，没办法快速高效的拓展配置套件能力。石墨套件理想情况下的分层应该如下图，基于这个分层架构可以围绕石墨核心的套件能力封装出多样化的产品形态。




2025 年套件的主要优化目标应集中在以下两点：

* 丰富套件对外接口，为极速 SDK 和标准 SDK 提供更强大的能力

* 将不属于套件核心功能的模块和组件代理到外层，为多样化的垂直场景提供可拓展能力


基于上面的架构，从前端视角看极速版 SDK 和标准 SDK 其实是同一套代码，只是套件底层能力不同形态的封装，研发迭代路径应该为：

1. 基于套件现有能力封装出极速 SDK

2. 基于极速 SDK 的项目架构重构标准 SDK

3. 逐步将套件内的非核心功能提升到代理层，并丰富接口配置，参考：[https://solution.wps.cn/docs/](https://solution.wps.cn/docs/)
