## 背景

我们在2024年强调主推SDK，不要打海外，不要搞AI，多做定制化搞钱，做了大幅度的战略收缩。这是因为我们现金流有非常大的压力，在这种场景下做了大量产品和技术上侧重点调整。


经过一年多的调整。我们当前环境又产生了很多变化，因此要重新考虑未来的规划问题。

第一点，最重要的是z项目，有一笔资金，在短时间内，我们是安全的，我们不必在像过去一样，过于着急现金流，做一些我们并不想做对石墨没有太大意义的需求。


第二点，参考这个代销的目的文档 [https://shimo.net/wiki/5q50r5WOywiDr13D](https://shimo.net/wiki/5q50r5WOywiDr13D)，我们引入了新创团队。这种组织架构的变化，对于石墨至关重要。根据康威定律，这种组织架构变化，使得我们技术架构大幅度调整，从技术角度上，能够把服务精简，并且在项目管理上能够更好的拉分支合版本，能够做到同时支撑多个客户需求，更适合商业化的发展。过去产研侧会害怕商业化大范围推广石墨文档的全站版本，这个部署和运维都极其复杂，因此在一定程度上只让商业化去推SDK这类产品，因为站在当时角度SDK的成本低，那么利润高，过去的全站成本高，利润低。而这次改变，使得 Drive 可以替换原来的复杂全站架构，大幅度降低成本，利润是可观的。因此后续商业化是可以大范围去推Drive这类产品。并且我们有个信心，我们可以是文档届的gitlab，让大部分人都能用上私有化的石墨文档。


第三点，俄罗斯项目UCCP的开启，我们有一些新的机会。能让我们做成一个类飞书的产品，能让我们有机会，做应用市场和海外石墨。并且从技术架构上应该多考虑国际化，避免挤牙膏式的做国际化。然后该项目，可以让我们有机会打海外google doc私有化的场景。同时这个应用市场，如果可行，未来可能海外AI项目可以直接接入进来，和石墨文档联动。


第四点，自动化部署和运维工具在co1.5版本真正意义上的实现，为什么这么说，是因为之前石墨的部署，升级，成本太高，需要SRE各种手动介入，核心原因是架构未参考业界方案。在co1.5做了重构。这个目的，是为了解决两个问题。

* 能够让非SRE人员，或者互联网上的人低成本的去部署，升级，运维。

* 以前说POC，都需要立项才能搞。现在非离线部署，都可以不需要立项，就能够POC部署。


第五点，稳定性

* Saas 监控报警：是重中之重。目前来看后端加入到监控，报警多。但还缺前端的监控，报警。

* Saas 项目管理流程：加入发版流程，缺绩效管理。

* 日志：CO1.5加入日志

* 自动化巡检

* 稳定性不解决，铺量会带来非常大的运维成本


## 技术路线演进

* 目前收到了很多商业化的反馈

   * 专业文档功能不齐全

   * 专业文档JS SDK能力不行

   * 幻灯片不支持H5编辑

   * 流程图能力不行

   * 等等


这里面有两条技术路线

* 补齐 JS API 能力

* 补齐 套件功能 能力


### 组织架构

* Drive 最外层

* SDK 最外层 （后端+少量前端，胶水层）

* SDK Core 纯套件 （底层后端和底层前端，导入导出）

* 优点：

   * 按业务划分，更适合业务

   * 套件能力继续下层，去掉所有接口调用的依赖，纯JS项目，能够让套件能更关注于套件的UI库，JS的接口定义，更适合做出JS SDK API

   * SDK最外层，统一一些前后端逻辑，例如协同编辑，上传组件，并能够在私有化的对接过程中前后端是一个团队来处理。

* 缺点：

   * 对底层改造成本高



## 摘要与核心战略目标

基于当前现金流压力缓解、组织架构调整（引入新创团队）、技术架构重构（Drive 替代复杂全站架构）、国际化机遇（UCCP 项目）以及自动化运维能力提升等背景，石墨文档将在 2026 年进行一次全面的战略重塑。

核心战略目标：

1.商业化目标： 成为“文档界的 GitLab”，实现石墨文档私有化部署的大范围推广，以 Drive 产品为核心，实现可观的利润增长。

2.产品目标： 补齐套件功能短板，实现 SDK/JS API 能力的全面增强，构建以 Drive 为核心、SDK 为开放接口、AI 为赋能引擎的标准化产品矩阵。

3.组织与技术目标： 落地“组织架构 2”（Drive/SDK/SDK Core）以支撑业务发展，完成核心技术栈重构（协同编辑 Go 化、轻文档重构），并实现产品国际化和高可用性。

## 一、 战略背景与驱动力分析

|驱动力|核心变化点|战略影响|
|:----|:----|:----|
|资金与现金流|Z 项目资金到位，短期内现金流安全。|从“求生存”转向“求发展”，减少低价值定制化需求，聚焦长期战略。|
|组织架构调整|引入新创团队，组织架构向“康威定律”驱动的技术架构调整。|技术架构精简，项目管理更灵活，能同时支撑多客户需求，更适合商业化。|
|产品架构重塑|Drive 替换原有复杂全站架构，大幅降低部署和运维成本。|商业化可大范围推广 Drive 类产品，利润可观，实现“文档界的 GitLab”愿景。|
|国际化机遇|俄罗斯 UCCP 项目开启，提供类飞书产品、应用市场和海外私有化场景机会。|推动技术架构国际化，避免“挤牙膏式”国际化，抢占海外 Google Doc 私有化市场。|
|运维能力提升|CO1.5 实现自动化部署和运维工具。|大幅降低部署、升级、运维成本，实现非 SRE 人员低成本部署，加速 POC 落地。|
|稳定性挑战|前端监控、绩效管理、自动化巡检等仍是短板。|稳定性是铺量推广的前提，必须优先解决，否则将带来巨大的运维成本。|

## 二、 组织架构与技术路线选择

### 1. 组织架构：采纳“组织架构 2”

选择： 采纳组织架构 2（Drive + SDK + SDK Core）作为核心组织架构，并以 War Room 方式解决组织架构 3 的快速功能补齐问题。

|组织架构 2 结构|职责定位|优势分析|
|:----|:----|:----|
|Drive|最外层，负责 Drive 产品的整体体验、标准化和商业化落地。|聚焦 Drive 核心场景和卖点，推动标准化和可视化配置部署。|
|SDK|最外层（后端 + 少量前端，胶水层），负责统一前后端逻辑（如协同编辑、上传组件）和私有化对接。|统一私有化对接团队，提升交付效率和稳定性。|
|SDK Core|纯套件（底层后端和底层前端，导入导出），负责套件的纯 JS 化和核心能力建设。|核心优势： 套件能力下沉，纯 JS 项目，专注于 UI 库和 JS API 定义，更适合打造高可用的 JS SDK API。|

### 2. 技术路线：补齐 JS API 能力与套件功能

两条技术路线并行，但以补齐 JS API 能力为核心，以补齐套件功能为保障。

|技术路线|核心目标|关键行动（P0/P1）|组织支撑|
|:----|:----|:----|:----|
|补齐 JS API 能力|打造高可用、开箱即用的 SDK，提升开发者体验。|1. 建设 SDK 文档、教程、FAQ 体系。 2. 提供 Java/Go/PHP/JS 等多语言对接 SDK 依赖包。 3. 兼容协作和单人编辑 SDK JSAPI。 4. 持续扩充场景化开放能力 API。|SDK Core 团队|
|补齐套件功能|解决商业化反馈的短板，提升产品力。|1. 轻文档重构并灰度放量。 2. 补齐轻文档功能（分栏、高亮块、Block 块等）。 3. 持续完善表格、应用表格套件能力（高级权限、Workflow）。 4. 推进幻灯片、流程图套件建设。|Drive 团队 / War Room 机制|

## 四、 关键保障措施

### 1. 稳定性与高可用性

•技术重构： 协同编辑从 Node.js 切换到 Go 语言（RichDoc, MoDoc, 所有套件）。

•监控体系： 补齐前端监控和报警，完善后端监控。

•自动化： 落地自动化巡检、兼容性自动化测试（针对不同客户配置）、前端自动化测试、性能测试自动化。

•错误码规范： 统一前端和后端的错误码，优化文案，避免用户恐慌。

### 2. 商业化赋能与流程优化

•产品形态标准化： 明确标准产品形态（Drive、SDK），输出产品矩阵、功能清单、统一边界、版本和授权方式，供销售报价。

•销售赋能： 输出结构化方案（PPT + 案例模板），搭建 Demo 演示环境（Demo 平台 + 操作手册）。

•项目管理： 完善 Saas 项目管理流程，将绩效管理纳入发版流程。
