* 内部体验机制（Dogfooding）
  * 核心痛点：脱离客户视角
    * shimo.im 和 SDK，产品架构上并不是统一，没有真正自己体验私有化的全站和SDK产品
    * 过去研发团队缺乏对私有化全站和 SDK 产品的真实使用体验，导致对客户反馈的痛点重视不足，响应滞后。
  * 变革举措
    * 引入 Drive 团队：作为 SDK 的首个“内部客户”，在开发阶段即暴露集成与体验问题。
    * 构建 shimo.net：全员使用 Drive 和 SDK 产品，将私有化升级作为内部日常办公环境。
    * 真实环境暴露：通过私有化升级方式处理shimo.net Drive，把私有化问题暴露到内部使用环境。
* 流程变革：关键发现与成效
  * 关键发现：客户提的问题都是对的
    * 例子：协作头像显示对接API复杂
    * 例子：搜索对接API复杂
    * 客户反馈的时候，我们没有重视。当我们自己Drive对接SDK的时候，发现确实是个问题。
  * 架构与客户环境一致才能解决问题
    * 更早发现问题：在交付前拦截潜在缺陷
    * 提前优化体验问题：有些不应该属于定制化，主动处理
  * 从“被动修补”到“主动优化”的思维转变

* 研发架构
  采用"Drive + SDK + SDK Core"三层架构：

| 层级 | 职责 | 优势 |
|------|------|------|
| **Drive** | 最外层，负责产品体验和商业化 | 聚焦核心场景和卖点 |
| **SDK** | 胶水层，统一前后端逻辑 | 统一私有化对接团队 |
| **SDK Core** | 纯套件，专注UI库和JS API | 打造高可用JS SDK API |
