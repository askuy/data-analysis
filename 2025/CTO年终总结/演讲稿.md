# 2025年度研发中心工作汇报 - 演讲稿

---

各位领导、同事，大家好！

我是研发中心负责人，今天向大家汇报2025年度研发中心的工作。

---

## 一、2025年，我们做成了什么？

先用四句话概括今年的核心变化：

第一，Drive 战略落地。我们从零组建团队，半年交付三大标杆客户，大幅降低部署运维和开发成本。

第二，内部体验机制建立。我们构建了 shimo.net，推行 Dogfooding，站在客户视角发现并解决真实痛点。

第三，自动化部署能力。实现在线模式 POC 部署无需立项，非 SRE 人员也能完成，极大提升交付效率。

第四，AI 提效。一个月实现类飞书产品，AI 参与工时评估、用户报告、测试、部署工具等全流程工作。

这四个变化背后，有一条贯穿全年的主线，就是 Drive 战略。接下来，我围绕这条主线展开汇报。

---

## 二、为什么要做 Drive？

我们的愿景是打造"文档界的 GitLab"。

回顾2024年，我们面临三个痛点：全站版本部署和运维复杂；成本高、利润低；只能支撑小功能定制化。

到了2025年，通过 Drive 架构：部署大幅简化；成本降低、利润可观；能够支撑大功能定制化。

具体来说，2025年我们交付了广东电信、好未来、OPPO 三大定制化客户，完成了所有者功能、密级标签系统、整个 Header 和 Sidebar 的 JS SDK、群组功能、折叠屏适配等大功能定制化。

一句话总结：Drive 替代复杂全站架构，让商业化可以大范围推广。

---

## 三、Drive 战略如何落地？

战略要落地，必须靠交付。

今年我们完成了17个大版本升级项目、13个正式交付项目、15个 POC 试点项目。我们在 TOP 10 客户上投入大量工时，确保交付质量。

三个标杆案例说明我们的交付能力：

广东电信，数据迁移耗时从6到8小时缩短至30分钟。

好未来，成功完成大规模复杂环境的灰度迁移。

OPPO，完成极具挑战性的双环境同步迁移。

---

## 四、交付能力靠什么支撑？

靠三大技术突破。

第一个突破是自动化发布和升级。

2024年广东电信发布的时候，需要比照代码配置，耗时3天，SRE 手动操作。到了2025年，三大客户定制化发布，直接拿包，SRE 和研发均可操作。

同时我们建设了日志系统，客户侧问题能够尽快发现。工具的可视化，极大降低了运维成本。POC 部署变得轻量化、去专业化。

第二个突破是应用表格性能。

应用表格实现了从"无法打开"到"4.5秒打开"的质变。我们支持了百万单元格；高级权限系统全面上线；协作模块完成底层统一。

第三个突破是极速 SDK 统一。

我们完成了前端入口统一，消除集成差异，降低私有化对接成本。

---

## 五、稳定性做得怎么样？

先看数据。Bug 数逐月下降，从年初的高峰持续回落。SaaS 接口稳定性达到 99.99%。导入导出稳定性从年底目标的 99.1% 提升到了 99.9%，这个我们做到了。

我们做了什么？

值班制度常态化。报警配置了150条。重构后的业务报警配置是大于0就报警，老业务正在迁入，目前是大于10。

同时我们编写了大量 SOP 文档。带来的效果是导入导出工单变少了，人员可以更多地做需求，而不是繁琐地对接工单 Bug。

这里我想说一个观点：不是客户不想报 Bug，而是我们因为 Bug 太多，要解决被客户问题拖累的局面。短期内靠交付团队和 SOP；长期内靠质量稳定，包括 AI 辅助的单元测试、自动化测试，以及 SaaS Error 日志大于0就报警。

我们实现了从"被动救火"到"主动防火"的转变。

---

## 六、Dogfooding 机制解决了什么问题？

稳定性解决的是"能不能用"的问题，Dogfooding 解决的是"好不好用"的问题。

先说核心痛点。过去 shimo.im 和 SDK 产品架构上并不统一，我们没有真正自己体验私有化的全站和 SDK 产品。研发团队缺乏对私有化产品的真实使用体验，导致对客户反馈的痛点重视不足，响应滞后。

我们做了什么变革？

引入 Drive 团队作为 SDK 的首个"内部客户"，在开发阶段就暴露集成与体验问题。构建 shimo.net，全员使用 Drive 和 SDK 产品，将私有化升级作为内部日常办公环境。通过私有化升级方式处理 shimo.net，把问题暴露到内部使用环境。

关键发现是什么？客户提的问题都是对的。

举两个例子。协作头像显示对接 API 复杂；搜索对接 API 复杂。客户反馈的时候，我们没有重视。当我们自己 Drive 对接 SDK 的时候，发现确实是个问题。

架构与客户环境一致才能解决问题。我们能更早发现问题，在交付前拦截潜在缺陷；能提前优化体验问题，有些不应该属于定制化，主动处理掉。

这是从"被动修补"到"主动优化"的思维转变。

---

## 七、AI 提效做了什么？

我想强调一个观点：使用 AI 不是简单与 AI 对话，而是掌握 Skill、MCP、Agent 等高级技能。

海外类飞书项目，1个月做完。古法编程不可取，无论如何先要用 AI 试试。AI 可以前后端一起调试，包括 AI 自己登录、自己探测浏览器报错、探测数据库数据是否准确。

大项目、定制化项目，给 CSV，AI 可以直接预估工时。

测试用例可以直接用 AI 转成 Playwright 脚本。

Word 预览重构，Opus 能够结合 MCP，直接理解 Word OOXML 协议，快速提升我们的新版 Word 预览。现在已经 OK，比老石墨增加更多排版能力和准确性。

---

## 八、整体架构是什么样的？

研发中心采用"Drive + SDK + SDK Core"三层架构。

最外层是 Drive，负责产品体验和商业化，聚焦核心场景和卖点。

中间是 SDK，是胶水层，统一前后端逻辑，统一私有化对接团队。

最底层是 SDK Core，是纯套件，专注 UI 库和 JS API，打造高可用 JS SDK API。

---

## 九、我们需要反思什么？

有三个问题需要反思。

第一，客户反馈重视不够。Dogfooding 后验证了客户是对的。

第二，代码问题占比高。Bug 的 69.3% 为代码问题。

第三，人员单点风险。每个研发仅负责一个套件。

同时我们持续面临三个挑战。

专业文档功能不齐，功能覆盖度仍需提升。幻灯片 H5 编辑缺失，移动端不支持编辑。交付持续消耗，占总工时 20% 以上。

---

## 十、2026年，我们要做什么？

2026年，我们向着"文档界的 GitLab"坚定迈进。

围绕以客户为中心，我们聚焦五件事：

第一，Drive 战略落地，验证架构演进。

第二，体验机制建立，回归客户视角。

第三，自动化部署，赋能商业推广。

第四，产品层面，补齐套件功能短板，SDK 和 JS API 全面增强。

第五，技术层面，增强稳定性，提升性能。

---

## 结语

回顾2025年，我们围绕 Drive 战略这条主线：验证了架构演进的正确性；回归了客户视角的 Dogfooding；在 Drive 上投入重兵确保交付质量与口碑。

2026年，我们将继续以客户为中心，让石墨成为企业文档协作的首选。
